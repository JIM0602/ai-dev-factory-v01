## 1. 项目背景

### 1.1 行业现状

拼豆（Perler Beads / Fuse Beads）作为一种手作体验活动，近年来在社交平台（小红书、抖音）上热度持续走高，吸引了大量年轻消费者。目前市场上的拼豆店以个体经营为主，门店常设在公寓楼、商住楼等非沿街位置。

### 1.2 经营痛点

通过调研和需求分析，拼豆店个体商家普遍面临以下 5 个核心痛点：

| 序号 | 痛点 | 影响 |
|------|------|------|
| 1 | **到店扑空**：顾客好不容易找到门店却发现满座，体验极差，容易流失客源。 | 客诉率高、复购率低 |
| 2 | **位置难找**：门店在公寓楼内无临街招牌，找路过程消耗顾客耐心。 | 到店前即产生负面体验 |
| 3 | **多渠道预约混乱**：预约来自小程序、电话、微信私聊、直接到店等多种渠道，商家凭脑子记或纸质本子记，容易漏单、重复接单。 | 超接导致顾客无座、漏单丢失生意 |
| 4 | **计时靠人记**：到店计时和加时全靠商家记忆，人多时容易混乱，记错谁还剩多久、谁该加钱。 | 纠纷风险、顾客信任降低 |
| 5 | **会员记录缺失**：回头客无法被识别，消费偏好和历史无从追溯，复购数据无法沉淀。 | 错失复购机会、难以做精准营销 |

### 1.3 解决方案概述

本项目通过**双端微信小程序**（用户端 + 商家端）配合**商家 Web 后台**，为拼豆店个体商家提供一套轻量、开箱即用的预约、容量管理、到店登记、计时管理和会员记录工具。

## 2. 产品目标

### 2.1 商业目标

- **降低顾客到店扑空率**：通过实时容量展示和预约确认机制，让顾客预约后才能确保有位置，而非盲目到店。
- **提升桌位利用率**：通过系统化的预约和计时管理，减少空置时段，提高翻台效率。
- **提升商家运营效率**：将预约处理、到店登记、计时管理等操作从"人脑记忆"升级为"系统记录"，减少人工失误。

### 2.2 用户目标

**顾客侧**：
- 打开小程序就能了解门店环境、位置、营业信息。
- 3 步以内完成预约（选日期、选时段、提交）。
- 随时查看自己的预约状态，需要时可自行取消。
- 到店路径清晰，知道怎么找到门店。

**商家侧**：
- 手机上随时接收和处理预约，不需守在电脑前。
- 一眼看清当前桌位占用情况和每个顾客的剩余时间。
- 电话/微信预约也能统一录入系统，不遗漏。
- 回头客信息自动记录，消费历史一目了然。

## 3. 产品边界

### 3.1 V1 包含功能（本期必做）

**模块一：门店展示与配置**

| 功能 | 说明 | 所在端 |
|------|------|--------|
| 门店主页展示 | 展示门店照片轮播、店名、地址、营业时间、剩余容量，底部有"预约"入口按钮 | 用户端小程序 |
| 门店信息配置 | 商家在 Web 后台设置门店名称、地址、门牌号、照片、营业时间、桌位/工位总数 | 商家 Web 后台 |

**模块二：顾客预约**

| 功能 | 说明 | 所在端 |
|------|------|--------|
| 发起预约 | 顾客选择日期、时段、人数后提交预约 | 用户端小程序 |
| 容量校验 | 提交预约时系统自动校验所选时段是否还有空位，满了提示顾客换时段 | 后端逻辑 |
| "预约需确认"开关 | 商家在后台设置：开启后顾客预约需商家确认才生效；关闭后预约自动确认 | 商家 Web 后台 |
| 预约确认/拒绝 | 商家在商家端小程序或 Web 后台对"待确认"预约进行确认或拒绝 | 商家端小程序 + Web |
| 预约状态查看 | 顾客在"我的预约"中查看所有预约及状态（待确认/已确认/已取消/已拒绝/已完成） | 用户端小程序 |
| 顾客取消预约 | 顾客可在预约时间前自行取消已确认的预约 | 用户端小程序 |
| 自动取消 | 若开启"预约需确认"，预约时间前 X 小时仍未确认的预约自动取消，释放容量 | 后端定时任务 |

**模块三：商家手动添加预约**

| 功能 | 说明 | 所在端 |
|------|------|--------|
| 代客预约 | 顾客通过电话/微信联系商家，商家在系统内填写顾客姓名、手机号、日期、时段、人数，直接生成已确认预约 | 商家端小程序 + Web |
| 容量校验 | 手动添加同样受容量限制，超容量时提示商家 | 后端逻辑 |

**模块四：到店登记与团购券核销**

| 功能 | 说明 | 所在端 |
|------|------|--------|
| 到店登记 | 顾客到店后，商家在预约列表中找到该顾客记录，点击"到店登记"，记录到店时间，并开始计时 | 商家端小程序 |
| 团购券核销登记 | 到店登记时可选择录入团购券码（第三方平台如美团、抖音售出的兑换券），系统记录核销信息 | 商家端小程序 |

**模块五：计时管理**

| 功能 | 说明 | 所在端 |
|------|------|--------|
| 计时看板 | 商家查看所有正在计时的顾客列表，每行显示：顾客姓名、桌位号、已用时长、剩余时长 | 商家端小程序 |
| 剩余时间提醒 | 剩余时间不足（如少于 15 分钟）时，该条记录高亮提示商家 | 商家端小程序 |
| 加时操作 | 商家点击"加时"，选择增加的时长（如 +30 分钟、+1 小时），系统更新剩余时长 | 商家端小程序 |
| 结束计时 | 顾客离店时，商家点击"结束计时"，记录离店时间，释放桌位 | 商家端小程序 |

**模块六：会员记录**

| 功能 | 说明 | 所在端 |
|------|------|--------|
| 消费记录自动沉淀 | 每次顾客完成计时（离店）后，系统自动记录：顾客姓名、手机号、到店日期、消费时长、是否使用团购券 | 后端自动 |
| 会员查询 | 商家按手机号或姓名搜索顾客，查看其历史到店记录 | 商家端小程序 + Web |
| 会员记录详情 | 展示顾客的累计到店次数、每次到店的日期、时长、券使用情况 | 商家端小程序 + Web |

**模块七：基础消息提醒（预留）**

| 功能 | 说明 | 所在端 |
|------|------|--------|
| 模板消息预留 | 后端准备好发送微信订阅消息的接口和数据字段，实际触发逻辑在 V1 预留但暂不完全启用 | 后端 |

### 3.2 V1 明确不做（范围控制）

以下功能在 V1 阶段**明确不做**，避免功能膨胀：

| 不做内容 | 原因 | 列入后续版本 |
|------|------|:---:|
| 在线支付 | V1 聚焦预约与计时管理，支付在线下完成 | V2 |
| 自动计费（按时长算钱） | 涉及不同时段定价、团购券折算等复杂逻辑，V1 商家手动计费 | V2 |
| 储值、次卡、积分、会员等级 | 属于会员营销体系，V1 先做基础记录 | V2/V3 |
| 候补预约 | 排队逻辑涉及自动通知、限时确认等复杂流程 | V2 |
| 多门店管理 | V1 仅支持单门店经营 | V2 |
| 复杂营销玩法（拼团、秒杀、优惠券） | 超出 V1 MVP 范围 | V3+ |
| 连锁权限体系（总部管理/分店/店员角色） | V1 仅支持单个商家账号，无角色分级 | V2 |
| 数据统计分析看板（图表、趋势） | V1 仅做基础记录查询，不做聚合分析 | V2 |

## 4. 用户角色详细描述

### 4.1 顾客（C 端用户）

**画像**：对拼豆手作感兴趣的消费者，典型场景包括朋友聚会、亲子互动、情侣约会等。年龄集中在 18-35 岁，女性占比偏高，习惯使用微信和小程序。

**行为特征**：
- 通过社交平台（小红书、抖音）或朋友推荐得知某家拼豆店。
- 首次使用时会查看门店环境照片，判断是否值得前往。
- 预约前最关心：有没有位置、怎么找到门店。
- 到店后可能因为玩得尽兴需要加时。

**核心场景**：
1. 周末想和朋友去玩拼豆，提前在小程序上看门店情况和预约。
2. 已经预约了周六下午，想查看预约是否已确认。
3. 临时有事去不了，需要取消预约。

**不需要的功能**：顾客不需要注册账号（通过微信授权即可），不需要在线支付（线下付款）。

### 4.2 商家（B 端用户）

**画像**：拼豆店个体经营者，通常自己就是店主，可能有一个兼职帮手。店内不一定有电脑，主要操作场景是**手机上随时处理事务**，同时也需要在电脑上做更细致的门店配置和历史记录查询。

**行为特征**：
- 早上到店后查看今天的预约情况，心里有数。
- 顾客打电话/发微信预约时，打开手机手动录入。
- 顾客到店时，在手机上快速登记并开始计时。
- 不忙时偶尔看看谁快到期了，主动询问是否需要加时。
- 晚上关店后可能在电脑上查看预约和会员记录。

**双端使用场景**：

| 场景 | 使用端 | 原因 |
|------|--------|------|
| 查看今日预约、处理预约 | 商家端小程序 | 移动中、随时处理 |
| 手动添加预约 | 商家端小程序 | 电话/微信预约时手边就是手机 |
| 到店登记 + 开始计时 | 商家端小程序 | 店里走动时操作 |
| 计时看板 + 加时操作 | 商家端小程序 | 随时瞄一眼谁快到期 |
| 门店信息初次设置 | 商家 Web 后台 | 信息量大，电脑配置更方便 |
| 预约规则调整 | 商家 Web 后台 | 不频繁操作，电脑设置更稳妥 |
| 预约记录筛选、查看历史 | 商家 Web 后台 | 屏幕大，列表浏览效率高 |
| 会员记录查询 | 两边都用 | 急查用手机，细看用电脑 |

### 4.3 系统管理员（V1 不做）

V1 版本只有一个商家角色，不区分管理员、店员等角色。管理账号只有一个，商家登录即可使用全部功能。

## 5. 功能需求（按模块详述）

### 5.1 门店展示模块

#### 5.1.1 门店主页（用户端小程序）

**功能描述**：用户打开小程序后看到的第一个页面，承载门店形象展示和预约入口的核心作用。

**页面构成**：
- **门店照片轮播**：顶部 Banner 区域，展示商家在后台配置的门店环境照片，支持左右滑动多图浏览。
- **门店名称**：突出展示，如"XX 拼豆工作室"。
- **门店地址**：文字展示完整地址（含门牌号、楼层、房间号），右侧有"导航"图标，点击可唤起微信内置地图并导航到该地址。
- **营业信息**：展示营业时间（如"周一至周日 10:00-22:00"），当前是否在营业中。
- **剩余容量**：展示"今日还可预约 X 个时段"，给顾客直观的空位感知。
- **门店介绍**（可选）：商家可在后台配置一段文字介绍，如"提供 200+ 种颜色拼豆，适合亲子、情侣、朋友聚会"。
- **底部固定预约按钮**：大而显眼的按钮，文案为"立即预约"，点击进入预约页面。如果当前不在营业时间或今日已满，按钮置灰并显示对应提示。

#### 5.1.2 门店信息配置（商家 Web 后台）

**功能描述**：商家在 Web 后台设置门店的基础信息，设置后自动同步到用户端小程序展示。

**可配置项**：
- 门店名称（必填，最长 30 字）
- 门店地址（必填，需包含省市区街道门牌号详细信息）
- 门店门牌指引（选填，如"3 楼出电梯右转 302 室"）
- 门店照片（必填，最少 1 张，最多 9 张，建议尺寸 750x500 以上）
- 营业时间：每日开始时间、每日结束时间（如 10:00 - 22:00）
- 休息日设置：支持设置每周固定休息日（如每周一休息），或设置特殊日期休息
- 桌位/工位总数（必填，数字，如"8 个桌位"）
- 门店介绍文案（选填，最长 200 字）

### 5.2 预约模块

#### 5.2.1 顾客预约（用户端小程序）

**功能描述**：顾客选择日期、时段、人数后发起预约，系统进行容量校验后创建预约记录。

**预约流程**：

1. **选择日期**：默认显示"今天"，可往后选择。预约可提前天数由商家在后台设置（默认 7 天）。过期的日期和商家休息日不可选。
2. **选择时段**：显示该日期可预约的时段列表。每个时段显示"开始时间 - 结束时间"和该时段剩余容量（如"10:00-11:00 余 3 位"）。已满的时段标灰不可选。时段粒度默认为 1 小时，具体时长由商家在后台配置。
3. **选择人数**：（可选，默认 1 人）如商家配置了"超人数需占用多个桌位"规则，则人数会影响容量计算。V1 简化：一个预约占用一个桌位，人数仅作参考，不影响容量。
4. **填写联系信息**：自动获取微信昵称和头像，手机号需用户手动填写（或通过微信授权获取）。
5. **提交预约**：点击"确认预约"按钮，系统校验：
   - 所选时段是否还有空余桌位
   - 如有空位，创建预约记录
   - 如商家开启了"预约需确认"，状态为"待确认"
   - 如商家关闭了"预约需确认"，状态直接为"已确认"
   - 如已满，提示"该时段已约满，请选择其他时段"

**预约规则**：
- 同一顾客相同时段不能重复预约。
- 预约成功后所占用的桌位/容量从该时段可用容量中扣减。
- 预约取消或被拒后，容量自动释放。

#### 5.2.2 我的预约（用户端小程序）

**功能描述**：顾客查看自己所有预约记录，按状态筛选，可进行取消操作。

**列表展示**：
- 按时间倒序排列，最近预约在最上面。
- 每条显示：预约日期、时段、状态标签（待确认/已确认/已取消/已拒绝/已完成）、操作按钮。
- 状态说明：
  - **待确认**（黄色）：等待商家确认
  - **已确认**（绿色）：预约成功，按时到店即可
  - **已取消**（灰色）：顾客自己取消了
  - **已拒绝**（红色）：商家拒绝了预约
  - **已完成**（蓝色）：已到店并已完成消费

**可操作项**：
- 点击某条预约进入详情页。
- "已确认"状态的预约，且当前时间在"预约时间 - 取消截止时间"之前，可点击"取消预约"。
- "待确认"状态的预约，可取消。
- "已取消"、"已拒绝"、"已完成"状态的预约仅可查看，不可操作。

#### 5.2.3 预约管理（商家端小程序 + Web 后台）

**功能描述**：商家查看所有预约记录，支持按日期和状态筛选，可对"待确认"预约进行确认或拒绝，也可手动添加预约。

**预约列表**：

| 列 | 说明 |
|----|------|
| 顾客信息 | 微信昵称 + 手机号 |
| 预约日期 | 格式：2026-05-11 |
| 预约时段 | 格式：14:00-15:00 |
| 人数 | 顾客填写的人数 |
| 状态 | 待确认 / 已确认 / 已取消 / 已拒绝 / 已完成 |
| 预约来源 | 顾客自行预约 / 商家代约 |
| 操作 | 确认/拒绝（待确认时）、到店登记（已确认时）、查看详情 |

**筛选条件**：
- 按日期：今天 / 明天 / 选择具体日期
- 按状态：全部 / 待确认 / 已确认 / 已完成 / 已取消
- 搜索：支持按顾客手机号或姓名搜索

**确认/拒绝预约**：
- 商家点击"确认"：预约状态变为"已确认"，顾客收到确认通知。
- 商家点击"拒绝"：可选填拒绝原因（如"该时段已满"），预约状态变为"已拒绝"，容量释放，顾客收到通知。

**手动添加预约（代客预约）**：
- 商家点击"添加预约"按钮。
- 填写：顾客姓名（必填）、手机号（必填）、预约日期（必填，日历选择）、时段（必填，下拉选择）、人数（选填，默认 1）。
- 提交时系统校验容量。如容量充足，直接生成"已确认"状态预约（不需确认流程）。
- 如容量不足，提示商家"该时段已满（当前剩余 X 个空位，需要 Y 个）"。

#### 5.2.4 预约规则配置（商家 Web 后台）

**功能描述**：商家在 Web 后台配置预约相关的规则。

**可配置项**：
- **预约需确认开关**：「开启」则顾客预约后需商家手动确认；「关闭」则顾客预约后自动确认。
- **提前预约天数**：顾客最多可提前多少天预约（如 7 天）。
- **预约截止时间**：距离时段开始前多久关闭预约（如"时段开始前 1 小时不可再预约"）。
- **自动取消规则**：当"预约需确认"开启时，距离预约时间前 X 小时仍未处理的"待确认"预约，自动取消。
- **顾客自主取消截止时间**：距离预约时间前 X 小时，顾客可以自行取消预约。过了这个时间，顾客不可自行取消，需联系商家。
- **时段时长**：每个预约时段的长度（如 60 分钟、90 分钟、120 分钟）。默认 60 分钟。

### 5.3 容量控制模块

**功能描述**：以"桌位/工位"为单位管理门店容量，每个时段最多可容纳的预约数不超过桌位总数。

**容量定义**：
- 门店总桌位数（如 8 个桌位）由商家在门店信息中设置。
- 每个时段可接收的最大预约数 = 门店总桌位数。
- V1 简化：一个预约 = 一个桌位，不考虑一个预约多人占用多个桌位的情况。

**容量校验时机**：
- 顾客预约提交时：校验所选时段剩余容量是否 >= 1。
- 商家手动添加预约时：同样校验容量。
- 预约取消/拒绝时：释放容量。
- 自动取消时：释放容量。

**容量展示**：
- 用户端门店主页显示"今日还可预约 X 个时段"（粗略感知）。
- 预约选时段页面显示每个时段的具体剩余容量（精确感知）。
- 商家端显示每个时段的已预约数/总数。

### 5.4 到店登记与团购券核销模块

#### 5.4.1 到店登记（商家端小程序）

**功能描述**：顾客到店后，商家在预约列表中找到该顾客的记录，进行到店登记，并开始计时。

**操作流程**：
1. 商家在预约列表中找到该顾客（可按手机号/姓名搜索，或直接浏览当日已确认预约）。
2. 点击该预约记录的"到店登记"按钮。
3. 系统弹出确认框，显示顾客信息，商家确认顾客已到店。
4. 同时展示"团购券核销"入口（可选，见 5.4.2）。
5. 商家确认后：
   - 记录实际到店时间（精确到分钟）。
   - 预约状态从"已确认"变为"已到店"或"计时中"。
   - 系统开始倒计时，初始时长为该预约的时段长度（如 60 分钟）。
   - 桌位从"空闲"变为"占用中"。
   - 该顾客出现在计时看板中。

**无预约到店（walk-in）处理**：
- 顾客没有预约直接到店，商家先走"手动添加预约"流程，创建一条"已确认"预约，然后立即进行到店登记。

#### 5.4.2 团购券核销登记（商家端小程序）

**功能描述**：顾客到店时出示团购券，商家录入券码做核销登记。V1 仅做记录，不验证券码真伪。

**操作流程**：
1. 在到店登记确认框中，选择"录入团购券"。
2. 手动输入或扫码录入券码（如美团券码、抖音券码）。
3. 可选择券来源（美团/抖音/其他，下拉选择）。
4. 可选择券类型（如"2 小时体验券"、"双人券"，下拉或手动填）。
5. 确认后，券码与本次消费记录关联。
6. 如顾客持多张券，支持录入多张。

**V1 不做**：不调用第三方 API 验证券码有效性。商家自行判断券码是否有效。

### 5.5 计时管理模块

#### 5.5.1 计时看板（商家端小程序）

**功能描述**：商家查看所有正在计时中的顾客，一眼掌握店内情况。这是商家使用频率最高的页面之一。

**看板展示**：
- 以卡片列表形式展示所有"计时中"的顾客。
- 每张卡片显示：
  - 顾客姓名（微信昵称）
  - 桌位号（如"3 号桌"，由商家在到店登记时分配或系统自动分配）
  - 预约时段（如"14:00-15:00"）
  - 已用时长（如"已过 32 分钟"）
  - 剩余时长（如"剩余 28 分钟"，用醒目的倒计时样式）
  - 剩余时间不足提醒（剩余 < 15 分钟时卡片变为橙色，剩余 < 5 分钟时变为红色）
- 按剩余时间从少到多排序，最快要到期的排在最上面。
- 顶部统计：正在计时 X 组、空闲桌位 Y 个。

**操作**：
- 卡片左滑或长按可出现操作菜单："加时"、"结束计时"。
- 或点击卡片进入详情页操作。

#### 5.5.2 加时操作

**功能描述**：顾客要求延长时间时，商家一键加时。

**操作流程**：
1. 在计时看板或顾客详情中点击"加时"。
2. 弹出加时选择器，可选：+30 分钟、+1 小时、+2 小时，或自定义输入分钟数。
3. 确认后，该顾客的剩余时长立即增加对应时长。
4. 卡片上的倒计时自动更新。

**记录**：加时操作记录在本次消费记录中，包括加时时长和操作时间。

#### 5.5.3 结束计时

**功能描述**：顾客离店时，商家结束计时并释放桌位。

**操作流程**：
1. 在计时看板或顾客详情中点击"结束计时"。
2. 弹出二次确认："确认 [顾客名] 已离店？计时将停止。"
3. 确认后：
   - 记录实际离店时间。
   - 计算本次消费总时长。
   - 顾客从计时看板中移除。
   - 桌位从"占用中"恢复为"空闲"。
   - 本次消费记录自动沉淀到该顾客的会员记录中。
   - 该时段的该桌位容量释放（如果有中途结束，该时段可被新预约使用，V1 简化为：结束即释放桌位，后续时段可重新预约）。

**V1 不做**：不自动计算费用。商家根据实际时长自行与顾客结算。

### 5.6 会员记录模块

#### 5.6.1 消费记录自动沉淀

**功能描述**：每次顾客完成计时（离店）后，系统自动将本次消费信息记录到该顾客名下。

**记录内容**：
- 顾客姓名、手机号（作为会员唯一标识）
- 到店日期
- 到店时间 - 离店时间
- 消费总时长
- 是否使用团购券、券码
- 预约方式（自行预约 / 商家代约 / walk-in）

**去重规则**：同一手机号视为同一位会员，多次到店记录归集到同一会员档案下。

#### 5.6.2 会员查询（商家端小程序 + Web 后台）

**功能描述**：商家按手机号或姓名搜索会员，查看其累计到店记录。

**搜索**：
- 支持手机号精确搜索和姓名模糊搜索。
- 搜索结果展示：会员姓名、手机号（脱敏显示中间四位）、累计到店次数、最近到店日期。

**会员详情**：
- 点击搜索结果的某条记录进入会员详情。
- 展示历史到店记录列表，按时间倒序排列。
- 每条记录显示：到店日期、消费时长、是否用券。
- Web 后台支持按日期范围筛选历史记录。

### 5.7 消息提醒模块（V1 预留）

**功能描述**：V1 阶段准备好消息通知的数据结构和接口，但暂不完全启用微信订阅消息的实际发送（取决于微信模板消息/订阅消息的申请审核状态）。

**预留的通知场景**：

| 通知场景 | 接收方 | 触发时机 |
|----------|--------|----------|
| 预约提交成功 | 顾客 | 顾客提交预约后 |
| 预约已确认 | 顾客 | 商家确认预约后 |
| 预约已拒绝 | 顾客 | 商家拒绝预约后 |
| 预约自动取消 | 顾客 | 系统自动取消后 |
| 预约即将开始 | 顾客 | 预约开始前 X 分钟 |
| 新预约通知 | 商家 | 有新预约提交时 |
| 计时即将到期 | 商家 | 顾客剩余时间不足 X 分钟时 |

**V1 实现方式**：
- 后端定义消息模板数据结构。
- 后端提供发送消息的接口（待接入微信订阅消息）。
- 数据库预留消息记录表。
- 前端不实现消息列表页面（V2 再做）。

## 6. 非功能需求

### 6.1 性能要求

| 指标 | 要求 |
|------|------|
| 小程序首页加载时间 | 不超过 3 秒（含图片加载） |
| 预约提交响应时间 | 不超过 2 秒 |
| 计时看板刷新 | 倒计时每 10 秒自动刷新一次，刷新响应不超过 1 秒 |
| Web 后台页面加载 | 不超过 3 秒 |
| 并发支持 | 同时至少支持 20 个顾客 + 3 个商家正常使用 |

### 6.2 兼容性要求

| 要求 | 说明 |
|------|------|
| 微信版本 | 支持微信 8.0 及以上版本 |
| 手机系统 | 支持 iOS 14+ 和 Android 8.0+ |
| 浏览器（Web 后台） | 支持 Chrome 90+、Edge 90+、Safari 14+ |

### 6.3 可用性要求

- 小程序核心操作（预约、查看状态）不超过 3 步。
- 商家高频操作（到店登记、加时）一键完成，无需多次跳转。
- 所有错误情况有明确的用户提示（而非系统报错页面）。
- 网络异常时有友好的离线提示。

### 6.4 安全性要求

- 商家账号通过微信登录授权，后端验证 OpenID。
- 顾客通过微信授权获取基本信息（昵称、头像），手机号通过微信授权获取或手动输入。
- 商家 Web 后台需要账号密码或微信扫码登录。
- 所有接口需验证身份，不允许未授权访问。
- 敏感数据（手机号）在前端做脱敏处理。
- API 请求使用 HTTPS。

### 6.5 数据要求

- 会员记录需持久化存储，不因版本更新丢失。
- 预约数据支持至少保留 12 个月，之后可归档。
- 计时数据实时更新，延迟不超过 2 秒。

## 7. 约束条件

### 7.1 技术约束

- 用户端和商家端基于微信小程序平台开发，需遵守微信小程序开发规范。
- 第一版按 CLAUDE.md 定义的技术栈开发：uni-app + Vue 3（小程序）、Vue 3 + Vite（Web 后台）、NestJS（后端）、PostgreSQL（数据库）。
- 部署采用 Docker Compose 方案。

### 7.2 业务约束

- V1 仅支持单门店单商家，不支持多门店和多角色权限。
- V1 不含在线支付，所有支付线下完成。
- V1 不含第三方团购平台 API 对接，券码仅做记录。
- 微信订阅消息的发送受微信政策限制，V1 仅做接口预留。

### 7.3 时间约束

- V1 目标是一个"可用的 MVP"，功能完整但交互可简化，后续迭代优化。
- 预计开发周期由后续阶段拆解任务后确定。

### 7.4 设计约束

- 小程序需遵循微信小程序的设计规范和交互模式。
- 商家端小程序和用户端小程序的视觉风格应统一（同一品牌）。
- Web 后台采用常规管理后台布局（侧边栏导航 + 主内容区）。

## 8. 关键假设（来自需求分析阶段）

以下假设在未与需求方确认前按此推进，标记为 H1-H10：

| 编号 | 假设内容 | 影响范围 |

| H5 | 单门店单商家账号，顾客无需注册 | 权限模型 |
| H6 | 商家端和用户端是同一个微信小程序，通过角色切换区分 | 项目结构 |
| H7 | 门店地址展示文字 + 可点击唤起微信地图，不集成复杂地图 SDK | 开发复杂度 |
| H8 | 倒计时到 0 后不自动结束，需商家手动操作 | 计时管理逻辑 |
| H9 | 顾客可在预约前 X 小时自行取消，X 由商家配置 | 预约规则 |
| H10 | 消息通知 V1 仅做接口和数据库预留，不完全启用 | 消息模块范围 |
