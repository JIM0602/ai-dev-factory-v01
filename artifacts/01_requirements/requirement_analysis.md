# 需求分析文档

## 项目名称

**拼豆店预约计时小程序**（暂定名）

**一句话描述**：面向拼豆店个体商家的双端微信小程序工具，核心解决门店容量管理、预约管理、到店计时和会员记录沉淀问题。

---

## 1. 核心解决的问题（用户痛点）

以下是当前拼豆店经营中真实存在的 5 个痛点：

| 序号 | 痛点 | 具体表现 |
|------|------|----------|
| 1 | 顾客到店后无空位 | 门店设在公寓楼内，顾客好不容易找到后却发现满座，体验极差，容易流失客源。 |
| 2 | 门店位置难找 | 拼豆店多在商住楼、公寓内，无临街招牌，顾客到店路径不清晰，大量时间花在找路上。 |
| 3 | 多渠道预约难以统一管理 | 顾客可能通过小程序、电话、微信私聊、直接到店等方式预约，商家靠脑子记或纸质本子记，容易漏单、重复接单。 |
| 4 | 计时与加时靠人工记忆 | 顾客到店后按小时计费，商家凭记忆计时，中途加时也靠口头约定。人多时容易搞混谁还剩多久、谁该加钱了。 |
| 5 | 会员和到店记录分散 | 没有系统记录顾客来过几次、什么偏好、消费多少，回头客无法识别，复购数据无法沉淀，不利于长期经营。 |

---

## 2. 目标用户角色及核心诉求

### 角色 A：顾客（C 端用户）

**画像**：对拼豆感兴趣的人，可能是朋友聚会、亲子活动、情侣约会等场景。通过小程序发现和预约门店。

| 核心诉求 | 说明 |
|----------|------|
| 快速了解门店 | 看到门店照片、环境、位置、营业信息，判断是否值得去。 |
| 顺利找到门店 | 小程序内能看到详细地址和导航指引，减少找路时间。 |
| 便捷预约 | 选择日期、时段、人数后发起预约，清楚知道是否预约成功。 |
| 清晰查看预约状态 | 随时查看自己的预约是"待确认"、"已确认"还是"已取消"。 |
| 到店后顺畅体验 | 到店登记、计时开始、中途加时等流程顺畅，不麻烦。 |

### 角色 B：商家（B 端用户）

**画像**：个体经营者，通常 1-2 人看店。店内不一定有电脑，更习惯用手机操作。但同时也有在电脑上做更细致管理的需求。

商家的使用场景分为两个端：

**商家端小程序（移动端，轻量操作）**：

| 核心诉求 | 说明 |
|----------|------|
| 随时处理预约 | 收到新预约时能收到通知，能快速确认或拒绝。 |
| 手动添加预约 | 顾客打电话或发微信预约时，商家人工录入系统，统一管理。 |
| 随时掌握剩余容量 | 一眼看到今天还有几个空位，避免超接。 |
| 到店登记 | 顾客到店后快速登记，自动开始计时。 |
| 计时看板 | 清楚看到每个在位顾客还剩多少时间，谁快到期了。 |
| 加时操作 | 顾客要续时，一键加时，自动更新剩余时间。 |

**商家 Web 后台（桌面端，细致管理）**：

| 核心诉求 | 说明 |
|----------|------|
| 门店信息配置 | 设置门店名称、地址、照片、营业时间、桌位数量等基础信息。 |
| 预约规则配置 | 设置是否需要商家确认、提前多久可预约、预约自动取消规则等。 |
| 预约记录管理 | 查看历史预约、筛选、导出等。 |
| 会员记录查询 | 查看每个顾客的历史到店记录、消费记录。 |
| 团购券核销查询 | 查看团购券核销记录，对账等。 |
| 数据看板 | 了解门店运营情况（预约量、到店率、翻台率等趋势），但不是 V1 重点。 |

## 3. 核心业务流程

### 流程 1：顾客预约（主线）

```
顾客打开小程序 → 查看门店主页（照片、地址、营业信息、剩余容量）
→ 点击"预约" → 选择日期、时段、人数 → 提交预约
→ 系统判断容量是否足够
  ├─ 容量不足 → 提示"该时段已满，请选择其他时段"
  └─ 容量足够 → 创建预约记录
      ├─ 商家开启了"预约需确认" → 预约状态为"待确认"，等待商家处理
      │   ├─ 商家确认 → 状态变为"已确认"，顾客收到确认通知
      │   └─ 商家拒绝 → 状态变为"已拒绝"，顾客收到拒绝通知
      └─ 商家关闭了"预约需确认" → 预约直接"已确认"
```

### 流程 2：商家手动添加预约

```
顾客通过电话/微信联系商家 → 商家打开商家端小程序
→ 点击"添加预约" → 填写顾客信息（姓名、手机号）、选择日期时段人数
→ 提交 → 直接生成"已确认"预约记录
```

### 流程 3：顾客到店 + 计时开始

```
顾客到店 → 商家在商家端小程序找到该顾客的预约记录
→ 点击"到店登记" → 登记到店时间
→ 如果有团购券 → 可选择"团购券核销"，输入券码
→ 确认开始计时 → 系统开始倒计时
→ 桌位状态从"空闲"变为"占用中"
```

### 流程 4：计时中 + 加时

```
商家打开"计时看板" → 看到所有正在计时的顾客列表
→ 每行显示：顾客名、桌位号、已用时长、剩余时长
→ 剩余时间不足提醒（如剩余 15 分钟高亮）
→ 顾客要求加时 → 商家点击"加时" → 选择加时时长（如 +1 小时）
→ 系统更新剩余时长
```

### 流程 5：预约自动取消

```
如果商家设置了"提前 X 小时未确认自动取消"：
系统定时检查 → 发现"待确认"预约距离预约时间不足 X 小时
→ 自动将预约状态改为"已取消" → 释放占用的容量 → 通知顾客
```

### 流程 6：会员记录沉淀

```
顾客每次完成到店消费后 → 系统自动记录：
- 顾客姓名、手机号
- 到店日期、消费时长、是否使用团购券
→ 商家可在 Web 后台按手机号或姓名查询会员的到店历史
```

## 5. 关键假设和需要确认的问题

### 我的合理假设（基于 brief.md 推断，未确认前按此推进）

| 序号 | 假设 | 理由 |
|------|------|------|
| H1 | 一个门店有固定数量的桌位/工位，每个桌位同时只能被一个预约占用。 | brief 提到"容量控制"和"桌位占用情况"，暗示以桌位为单位管理。 |
| H2 | 预约以"时段"为单位，比如 1 小时一个时段，顾客选择起始时段和预计使用时长。 | brief 提到"按时长经营"，合理推断预约需要选时段。 |
| H3 | 团购券是第三方平台（美团、抖音等）售出的兑换券，商家在本系统中录入券码即完成核销登记，不涉及券码生成和验证券码有效性。 | brief 说"团购券核销登记"，定位为"登记"而非"核销系统"，推断只是记录。 |
| H4 | 一个顾客可以预约多个桌位/工位（比如带朋友来），因此预约人数可能大于 1。 | 拼豆是社交活动，合理推断顾客会结伴。 |
| H5 | 商家账号只有一个（单门店单账号），顾客不需要注册账号，通过微信授权获取基本信息即可。 | brief 提到"个体商家"，且不含连锁权限体系。 |
| H6 | 商家端小程序和用户端小程序是同一个微信小程序，通过角色切换或登录身份区分界面。 | 双端合一是小程序常见做法，降低开发和维护成本。 |
| H7 | 门店照片和地址等信息由商家在 Web 后台配置，不需要地图 SDK 做复杂导航，只需要展示地址文字和可点击打开微信地图。 | V1 轻量原则，复杂地图集成可后续优化。 |
| H8 | 计时以"倒计时"形式展示剩余时间，到 0 后不自动结束，需要商家手动操作结束。 | 避免自动结束时顾客实际还在的尴尬。 |
| H9 | 预约取消规则：顾客可以在"预约时间前 X 小时"自由取消，商家可以在后台设置这个 X。 | 灵活性和防止恶意占位的平衡。 |
| H10 | 消息通知在 V1 阶段仅做"模板消息预留"，即后端接口和数据库字段准备好，但实际发送可能用微信订阅消息或不做，取决于微信政策。 | brief 明确写"基础消息提醒预留"。 |

### 需要和需求方确认的问题

| 序号 | 问题 | 为什么重要 |
|------|------|------------|
| Q1 | 预约的粒度是"桌位"还是"人数"？比如一个门店有 10 个桌子，每个桌子坐 2 人，那容量上限是 10 个预约还是 20 个人？ | 直接影响容量控制逻辑和数据模型设计。 |
| Q2 | 一个时段是多长？顾客预约时是选固定时段（如 10:00-11:00），还是选起始时间+使用时长？ | 影响预约交互和容量计算方式。 |
| Q3 | 团购券是否需要与本系统做 API 对接核销？还是纯粹手动录入券码做记录？ | 如果涉及 API 对接，开发量和复杂度完全不同。 |
| Q4 | 商家端小程序和用户端小程序是否合并在同一个小程序内？还是拆成两个独立小程序？ | 影响项目结构和用户进入路径。 |
| Q5 | 是否需要店员/员工角色？比如商家老板和兼职店员看到的界面不同？ | brief 说不含连锁权限，但单店也可能有店员。 |
| Q6 | 计时过程中，如果顾客提前离开，是否支持提前结束计时？计时费用是否按实际时长折算？ | 影响计时管理逻辑。 |
| Q7 | 顾客不预约直接到店（walk-in），商家如何操作？是先用"手动添加预约"再走正常流程，还是有一个快速开台功能？ | 影响商家操作流程设计。 |
| Q8 | 门店营业时间是否每天固定？是否支持设置每周不同的营业时间（如周一休息）？ | 影响预约可选时段的生成逻辑。 |

## 6. 成功指标建议

以下指标用于判断 V1 上线后是否达到预期效果。部分指标需要在上线后通过埋点和商家回访获取。

| 指标类别 | 具体指标 | 衡量方式 |
|----------|----------|----------|
| 用户体验 | 顾客从打开小程序到完成预约的时长 | 小程序埋点（V2 实现） |
| 用户体验 | 顾客"到店无空位"投诉率较上线前下降 | 商家反馈 |
| 商家效率 | 商家处理一个预约的平均耗时 | 商家回访 |
| 商家效率 | 计时混乱导致的纠纷次数下降 | 商家反馈 |
| 经营改善 | 桌位利用率（实际使用时长 / 营业时长） | 系统数据统计 |
| 经营改善 | 重复到店顾客占比 | 会员记录分析 |
| 系统稳定性 | 小程序无崩溃、无严重卡顿 | 微信后台监控 |
| 采用率 | 上线后 1 个月内至少有 1 家门店持续使用 | 系统活跃数据 |
