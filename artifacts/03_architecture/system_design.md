## 3. 各层职责

### 3.1 微信小程序 (uni-app)

- **顾客端（5 页）**：门店展示、预约提交、预约列表/详情、结果展示
- **商家端（8 页）**：工作台、预约管理、添加预约、计时看板、到店登记、会员查询/详情
- **角色切换**：同一 WeChat 用户，若 openid 匹配商家身份，可在小程序内切换顾客端/商家端视图
- **代码组织**：通过 `uni-app` 条件编译或页面路径前缀（`/pages/customer/` `/pages/merchant/`）区分两端
- **共享部分**：API 服务层、工具函数、Pinia Store 可共享

### 3.2 商家 Web 后台 (Vue 3 + Vite)

- **独立 SPA 项目**，不依赖微信环境
- **侧边栏布局**：工作台、预约管理、门店设置、预约规则、容量设置、会员管理、团购券记录
- **认证**：用户名 + 密码登录，JWT Token 存储于 localStorage
- **路由守卫**：未登录自动跳转登录页

### 3.3 后端服务 (NestJS)

- **认证**：微信登录（OpenID）+ Web 账密登录，统一签发 JWT
- **业务逻辑**：预约容量校验、状态流转、计时计算
- **定时任务**：每分钟扫描待确认预约，超时自动取消
- **API 风格**：RESTful，JSON 格式
- **错误处理**：统一异常过滤器，返回标准错误格式
- **日志**：请求日志 + 业务关键操作日志

### 3.4 数据库 (PostgreSQL)

- 存储所有业务数据
- 通过 TypeORM Migration 管理表结构变更
- JSON 字段用于灵活配置（如门店照片列表、休息日列表）

## 4. 通信方式

### 4.1 小程序 <-> 后端

```
[小程序] --- wx.request (HTTPS) ---> [Nginx:443] --- HTTP ---> [NestJS:3000]
```

- **认证方式**：请求头 `Authorization: Bearer <JWT>`
- **JWT 获取**：调用 `/api/auth/wechat-login` 传入微信 `code`，后端换取 OpenID 后签发 JWT
- **Token 刷新**：JWT 有效期 7 天，过期前通过 `/api/auth/refresh-token` 续期
- **角色标识**：JWT Payload 中包含 `role: "customer" | "merchant"`，后端 Guard 校验

### 4.2 Web 后台 <-> 后端

```
[浏览器] --- axios (HTTPS) ---> [Nginx:443] --- HTTP ---> [NestJS:3000]
```

- **认证方式**：请求头 `Authorization: Bearer <JWT>`
- **JWT 获取**：调用 `/api/auth/admin-login` 传入用户名密码，验证后签发 JWT
- **Token 存储**：localStorage，页面刷新后从 localStorage 恢复
- **角色标识**：JWT Payload 中 `role: "merchant"`，后台所有接口需要此角色

### 4.3 微信服务端 API 调用

```
[NestJS] --- HTTPS ---> [微信开放平台]
```

- **用途**：通过 `code` 换取 OpenID 和 Session Key
- **调用时机**：用户在小程序端调用 `wx.login()` 获取 code 后传给后端
- **依赖包**：`axios` 直接调用微信接口，无需额外 SDK

### Docker Compose 服务清单

| 服务 | 镜像 | 端口映射 | 说明 |
|------|------|----------|------|
| nginx | nginx:1.25-alpine | 443:443 | 反向代理 + 静态资源 |
| server | node:20-alpine (自定义) | 3000 (内部) | NestJS 应用 |
| postgres | postgres:15-alpine | 5432 (内部) | 数据库 |

### 数据持久化

- **PostgreSQL 数据**：Docker Volume `pgdata`，映射到宿主机 `/data/postgres`
- **上传文件**（门店照片）：Docker Volume `uploads`，映射到宿主机 `/data/uploads`，由 Nginx 直接提供静态文件服务

## 6. 微信小程序配置

### 6.1 小程序注册

- 1 个微信小程序 AppID，同时承载顾客端和商家端。
- 商家识别：后端数据库 `merchant_accounts` 表存储商家的微信 OpenID，小程序启动时调用 API 获取当前用户的角色。

### 6.2 小程序分包策略

V1 页面数量少（共 13 个小程序页面），无需分包。所有页面放在主包中即可满足 2MB 限制。

如需分包（未来扩展），可按顾客端和商家端拆分为两个分包：
- 主包：门店主页 (P01)
- 分包 A (customer)：P02-P05
- 分包 B (merchant)：B01-B08

## 7. 关键架构决策

| 编号 | 决策 | 理由 | 影响 |
|------|------|------|------|
| AD-01 | 顾客端和商家端共用一个 uni-app 项目 | 减少项目数量，共享组件和 API 层 | 页面路由需区分 `/pages/customer/` 和 `/pages/merchant/` |
| AD-02 | 商家端通过 openid 匹配识别，非独立登录 | V1 单商家场景，简化认证流程 | 商家需先在小程序端登录，后端手动录入其 openid |
| AD-03 | 时段容量通过 COUNT 查询实时计算，不建独立容量表 | 避免数据冗余和一致性问题 | 高峰期可能有轻微查询性能影响，V1 数据量小可忽略 |
| AD-04 | 计时看板倒计时由后端计算 + 前端定时轮询 | 简单可靠，不依赖 WebSocket | 10 秒轮询一次，网络开销小 |
| AD-05 | Web 后台使用 Element Plus 组件库 | 加速开发，管理后台交互标准化 | 包体积较大，但 Web 后台不限体积 |
| AD-06 | JWT 中存储角色信息，后端 Guard 统一鉴权 | 无状态认证，无需服务端 Session | JWT 泄露风险，需配合 HTTPS 使用 |
| AD-07 | V1 不做消息推送的实际发送 | PRD 明确 V1 仅预留 | 消息表建好，API 接口预留，V2 接微信订阅消息 |
| AD-08 | 门店照片存储到本地文件系统 + Nginx 静态服务 | V1 简单方案，避免引入 OSS | 需备份上传目录 |
