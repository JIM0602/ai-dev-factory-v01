## 1. 本次开发的亮点

### 1.1 流程完整性高

本项目的 AI Dev Factory v0.1 流程**首次完整跑通**。从 `brief.md` 的大白话需求开始，经过 9 个阶段，最终产出了：
- 完整的产品需求文档（需求分析 + PRD + 页面清单 + 用户流程 + 验收标准）
- 系统架构设计（架构图 + 数据库设计 + API 设计 + 权限规则 + 项目结构）
- UI Contest 竞标与融合方案（4 方案竞标 → 评选 → 融合 → 产出 Design Tokens + 组件规范）
- 56 个开发任务清单（含依赖关系和关键路径）
- 3 个端的完整代码（NestJS 后端 76 文件、小程序 50 文件、Web 后端 37 文件）
- 65 条单元测试（100% 通过）
- 代码审查报告（发现并修复 2 个 P0 阻塞问题）
- 测试报告（14 条功能流程验证）

### 1.2 阶段门控真正发挥作用

流程中最有价值的机制是**阶段之间的门控**：
- **需求阶段 → 产品阶段**：需求分析中的 10 个假设（H1-H10）和 8 个待确认问题（Q1-Q8）被带到 PRD 中，避免了"拍脑袋开发"。
- **产品阶段 → 架构阶段**：PRD 和验收标准为数据库设计、API 设计提供了明确的输入。
- **开发 → Review**：Review 发现了 2 个 P0 阻塞问题（计时看板倒计时计算错误、新会员数据丢失），这些问题如果跳过 Review 直接测试可能被遗漏。
- **Review → 测试 → 修复 → 再测试**：测试发现的问题通过 fix_list 跟踪，修复后再次验证，形成闭环。

### 1.3 UI Contest 机制产出高质量设计规则

UI Contest 是本次流程中的**最大亮点之一**。4 个方案（V01 温暖手作、V02 潮玩创意、V03 极简轻量、V04 自然森系）分别代表了不同的设计方向。最终融合方案以 V01 为主体，吸收：
- V03 的效率优化（Web 后台去暖色、形状+颜色双编码）
- V04 的暖棕调阴影、更宽的页边距
- V02 的按钮组交互（时段选择）

产出的 `final_design_rules.md` 包含了 12 个章节的完整 Design Tokens（颜色、字体、间距、圆角、阴影）、组件规范、布局规范、前端开发检查清单和禁止事项清单。这份文档让前端开发有了明确的"标准答案"，而非需要自行诠释设计意图。

### 1.4 后端代码质量扎实

NestJS 后端代码质量较高，体现在：
- **事务正确使用**：容量校验使用了 `SELECT ... FOR UPDATE` 悲观锁防止超卖
- **状态机逻辑正确**：预约状态流转（pending → confirmed → in_progress → completed）有明确的业务规则
- **测试覆盖关键路径**：65 条单元测试覆盖了 Auth、Reservation、Timer 三个核心服务的正常流程、异常分支和边界条件
- **统一错误码**：所有业务错误都有对应的错误码，前端可据此做精细化提示

### 1.5 多角色分工协作清晰

本次的 6 个开发角色（database-dev、backend-dev、miniapp-dev、merchant-web-dev、qa-tester、code-reviewer）各司其职，输出文件明确可追溯。

## 2. 遇到的问题和解决方案

### 2.1 P0 阻塞问题：计时看板倒计时计算错误

**问题**：后端 `getDashboard()` 未返回 `original_duration_minutes` 字段，前端直接用 `used_minutes` 代替 `original_duration_minutes` 计算总计时长。结果是：顾客原始预约 60 分钟，加时 30 分钟，实际应剩余 90 - 已流逝，但前端算出的是 60 - 已流逝，导致加时后倒计时不增加。

**根因**：前后端 API 契约不一致，后端遗漏了关键字段。

**解决**：后端在 `checkIn` 和 `getDashboard` 返回中增加 `original_duration_minutes`，前端改用 `original_duration_minutes + total_extension_minutes` 作为总时长。

**教训**：API 设计文档应明确列出响应体的每个字段。前后端开发应使用共享的 TypeScript 接口定义（DTO），避免字段遗漏。

### 2.2 P0 阻塞问题：新会员首次消费数据丢失

**问题**：`endSession()` 创建新会员时，先用 `memberRepo.create()` 创建空记录（`total_visits=0`, `total_duration_minutes=0`, `last_visit_date` 未设），再调用 `updateMemberStats()` 更新。但 `updateMemberStats()` 的逻辑是"累加"，而 create 时字段为 0/null，累加逻辑失效。结果：新会员首次消费后 `total_visits` 仍为 0，永久无法被搜索到。

**根因**：代码逻辑分散在两个方法中，create 和 update 各做一半，中间态不一致。

**解决**：将新会员的创建逻辑合并在 `endSession` 的事务回调中，create 时直接设置 `total_visits=1, total_duration_minutes=实际时长, last_visit_date=今天`。额外新增 2 条单元测试（T19/T20）防止回归。

**教训**：
- 创建+初始化不应分两步走。要么一次性设好初始值，要么确保 update 方法能正确处理初始空值。
- 关键数据变更必须有对应的单元测试覆盖新增/已有两种场景。

### 2.3 小程序构建环境缺失

**问题**：uni-app 项目依赖 `@dcloudio/vite-plugin-uni`，当前环境未安装，导致 `npx vite build` 失败。

**影响**：无法在本地构建和真机验证小程序。

**定级**：降级为 P2。这是开发工具链配置问题，不是代码缺陷。在正确环境（HBuilderX 或有完整 uni-app 依赖的环境）中可正常构建。

**教训**：项目 README 应明确列出所有开发依赖和安装步骤，包括 uni-app 工具链。CI 环境应预装 HBuilderX CLI 或完整的 uni-app npm 包。

### 2.4 Review 发现的其他值得关注的问题

- **Admin-web 主色系偏离设计规范**：Web 后台实际使用的 Element Plus 主题色与 `final_design_rules.md` 中定义的色值有偏差。建议 V2 做一次完整的主题色对齐。
- **小程序 BASE_URL 硬编码**：API 基础地址直接写死在代码中，部署时需手动修改。应改为环境变量或配置文件。
- **登录接口缺少限流**：`admin-login` 无速率限制，存在暴力破解风险。建议接入 `@nestjs/throttler`。

## 3. 流程改进建议（AI Dev Factory v0.1）

### 3.1 阶段输出应有更明确的"完成定义"

当前每个阶段的输出是文件。但文件质量差异较大。建议增加每个阶段的"最小检查清单"：

| 阶段 | 当前输出 | 建议增加 |
|------|---------|---------|
| 需求分析 | requirement_analysis.md | 需求方签字确认（或模拟确认） |
| 产品 | PRD + 页面清单 + 验收标准 | 验收标准的可测试性检查（每条 GWT 是否能被自动测试执行） |
| 架构 | system_design + db + api | API 响应体的字段级定义（非仅端点列表） |
| UI Contest | selected_ui_direction + final_design_rules | 前端开发者逐项确认设计规则可实现（消除 "设计很好但开发做不了" 的情况） |
| 计划 | task_list.md | 每个任务有明确的输出文件路径和验收命令 |

### 3.2 API 设计需要更"协议级"的细节

`api_design.md` 列出了所有端点及其入参/出参，但缺少：
- 每个响应字段的类型、是否可空、示例值
- 错误码的完整枚举
- 分页参数的统一约定（当前三端各写各的）

**建议**：API 设计阶段就产出 TypeScript interface/type 定义文件，既作为文档，又可被前后端直接 import 使用。这样 Review 阶段可以直接检查"接口定义 vs 实际返回"的一致性。

### 3.3 测试阶段应在开发中期就介入

当前流程是"全部开发完成 → 测试"。这导致 P0 问题在最后阶段才暴露。建议：
- **后端每个模块开发完就写单元测试**（本次已做到）
- **核心流程的集成测试在前后端联调阶段就执行**（本次未做到）
- **UI 合规检查在第一个页面完成后就做，而非等所有页面完成**

### 3.4 Review 应分多轮而非一次性

本次 Review 分了两轮（v1.0→v1.1→v1.2），第二轮聚焦修复验证。这种模式值得固化：
- **Round 1**：全量审查，产出问题列表
- **Round 2**：仅审查修复后的文件，验证修复质量
- **Round 3（可选）**：集成 Review，确认修复未引入新问题

### 3.5 增加"环境一致性"检查

本次的痛点之一是小程序构建环境缺失。建议在 `05_plan/` 阶段增加一个任务：
- **ENV-01**：确认开发环境、构建环境、测试环境的一致性
- 输出：`environment_setup.md`（列出每个端的依赖安装命令和验证命令）

## 5. 可复用经验

### 5.1 可直接复用的模块

| 模块 | 路径 | 复用场景 |
|------|------|---------|
| **Auth 模块**（微信登录 + JWT） | `server/src/modules/auth/` | 所有需要微信小程序登录的项目 |
| **Roles Guard** | `server/src/common/guards/roles.guard.ts` | 所有需要角色鉴权的 NestJS 项目 |
| **统一响应格式拦截器** | `server/src/common/interceptors/response.interceptor.ts` | 所有 NestJS 项目 |
| **JWT 双端认证策略** | `server/src/modules/auth/jwt.strategy.ts` | 同一后端服务多端不同 Token 有效期的场景 |
| **悲观锁防超卖** | `server/src/modules/reservation/reservation.service.ts` (createReservation) | 所有涉及库存/容量的预约系统 |
| **定时任务自动取消** | `server/src/modules/scheduler/` | 所有有"超时自动取消"需求的系统 |
| **StatusTag 双编码组件** | `miniapp/src/components/StatusTag.vue` | 所有需要无障碍状态标签的小程序 |
| **计时看板卡片** | `miniapp/src/components/merchant/TimerCard.vue` | 所有需要倒计时+状态分级的场景 |
| **Design Token 体系** | `final_design_rules.md` 第二章 | 可作为新项目的 Design Token 模板 |
| **验收标准 GWT 模板** | `acceptance_criteria.md` | 可作为同类产品的验收标准模板 |

### 5.2 可复用的流程模式

- **UI Contest 融合模式**：多方案竞标 + 评委评选 + 以主方案为主体吸收其他方案优点。适合所有 B2C 产品。
- **多端共享 Design Tokens**：颜色、字体、间距、圆角、阴影在后端、小程序、Web 三端使用相同的 Token 命名，仅单位不同（rpx vs px）。
- **阶段门控 + 不可跳过**：每个阶段有明确的输出文件和"不通过不可进入下一阶段"的硬约束。

## 6. 下次开发类似门店小程序的建议

### 6.1 产品层面

1. **"预约粒度"是第一个要明确的问题**：按桌位还是按人数？本次 V1 按桌位简化，但实际门店可能更关心人数。建议在需求分析阶段就与需求方确认。
2. **"时间段"设计比想象中复杂**：涉及到营业时间、休息日、时段时长、提前预约天数、取消截止时间等多个变量。建议先画一张"时间规则矩阵"再做数据库设计。
3. **Walk-in 流程要在一开始就设计好**：本项目中 walk-in 是通过"手动添加预约 → 立即到店登记"两步完成的，这是一种折中方案。如果 V2 优化，建议提供一键 walk-in。

### 6.2 技术层面

1. **计时场景优先考虑轮询而非 WebSocket**：V1 场景简单（单店 8 个桌位），10 秒轮询足够。V2 如果桌位数增加，再考虑 WebSocket。
2. **数据库行级锁是防超卖的最简单可靠方案**：`SELECT ... FOR UPDATE` 在事务中使用，不需要引入 Redis 或分布式锁。
3. **uni-app 的条件编译是管理双端小程序的关键**：顾客端/商家端共用一个项目，通过 `#ifdef` 和页面路径前缀区分。但需注意样式变量和 API 层是共享的。
4. **Element Plus 主题覆盖要趁早**：在项目搭建阶段就完整覆盖 Design Tokens，不要留到后面补。本次 Admin-web 的主题色偏差就是因为后期才做的事后覆盖。

### 6.3 流程层面

1. **在 DB-02（Entity）完成后就启动前端开发**：前端大部分任务（脚手架、组件、页面）不需要后端 API 就绪。并行开发可以大幅缩短周期。
2. **小程序的第一个页面完成后就做 UI 合规检查**：不要等所有页面都做完才检查。第一个页面就是规范样本。
3. **关键业务流程的测试用例在开发前就写好**（TDD）：本次的 14 条功能流程测试用例实际上来自验收标准。如果测试用例更早写好，开发就能以"通过测试"为目标编码。
